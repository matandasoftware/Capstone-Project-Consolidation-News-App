How POST Differs from GET:

1. PAYLOAD: POST sends 9-byte payload ("key=value"), GET sends 0-byte payload
2. HEADERS: POST includes additional headers:
   - Content-Length: 9
   - Content-Type: application/x-www-form-urlencoded
3. PURPOSE: GET retrieves data, POST submits data to the server
4. DATA LOCATION: GET data in URL, POST data in request body

Purpose of the Payload (key=value):

- Represents form data being submitted to the server
- Content-Type "application/x-www-form-urlencoded" tells server how to parse it
- Format: key=value pairs separated by & (if multiple)
- This is how HTML forms typically submit data to web servers

How the Payload is Transmitted:

- Sent in the HTTP request BODY (not in the URL)
- Content-Length header tells server exactly how many bytes to expect (9 bytes)
- Transmitted after all headers are sent
- More secure than GET (data not visible in URL/logs)

Server Response Body (IDENTICAL for both CURL and POWERSHELL):

The server reflects the data back showing:

- "method": "POST"
- "data": "key=value" - Raw body content as received
- "form": { "key": ["value"] } - Parsed form data
- "json": null - Not JSON format, so null
- "files": {} - No files uploaded
- Status Code: 200 OK - Data successfully received

--------------
Status Code: 404
Meaning: The requested resource does not exist on the server

Why might this occur?

- URL typo or incorrect path
- Resource was deleted or moved
- Incorrect routing/endpoint configuration
- Page or file never existed at that location

KEY TAKEAWAYS ABOUT HTTP

1. REQUEST/RESPONSE WORKFLOW:

   - Client sends request (method, headers, optional body)
   - Server processes and sends response (status code, headers, body)
   - All communication follows HTTP protocol structure
   - curl -v shows detailed TCP/TLS handshake and connection establishment
   - PowerShell -Verbose shows simplified request/response flow

2. METHODS HAVE DIFFERENT PURPOSES:

   - GET: Retrieve data (no body, idempotent, cacheable)
   - POST: Submit data (has body, non-idempotent, not cached by default)
   - HEAD: Get headers only (efficient status checking, curl uses -I flag)

3. HEADERS PROVIDE METADATA:

   - Request headers: What client sends (User-Agent, Accept-Encoding, Content-Type)
   - Response headers: What server sends (Date, Server, Content-Type, CORS)
   - Content-Type and Content-Length are critical for body transmission
   - Different tools send different User-Agent headers (curl/8.16.0 vs PowerShell/7.5.4)

4. STATUS CODES COMMUNICATE OUTCOMES:

   - 2xx: Success (200 OK)
   - 4xx: Client errors (404 Not Found, 403 Forbidden)
   - 5xx: Server errors (503 Service Unavailable)
   - Status code alone tells you if request succeeded
   - Both curl and PowerShell handle status codes identically

5. DATA TRANSMISSION:

   - GET: Data in URL query parameters (visible, limited size)
   - POST: Data in request body (more secure, larger size)
   - Form-encoded data uses Content-Type: application/x-www-form-urlencoded
   - Both tools transmit POST data identically (Content-Length: 9 for "key=value")

6. SECURITY & ACCESS CONTROL:

   - 403 indicates authorization issue (exists but denied)
   - 404 indicates resource doesn't exist
   - HTTPS encrypts transmission (port 443)
   - CORS headers control cross-origin access
   - TLS/SSL handshake visible in curl -v output

7. EFFICIENCY:

   - HEAD method retrieves only headers (no body)
   - Transfer-Encoding: chunked allows streaming (PowerShell)
   - Content-Length provides exact size (curl)
   - Accept-Encoding enables compression (gzip, deflate, br)

8. CURL vs POWERSHELL COMPARISON:

   - FUNCTIONALITY: Both produce identical HTTP requests and responses
   - VERBOSITY: curl -v shows more network details (DNS, TCP, TLS)
   - SYNTAX: curl has simpler syntax (-X, -d, -I flags)
   - OUTPUT: PowerShell provides structured objects, curl shows raw text
   - USER-AGENT: Different default User-Agent strings identify the tool used
   - HEADERS: curl shows lowercase headers, PowerShell uses Title-Case
